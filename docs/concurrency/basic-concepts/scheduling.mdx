# Scheduling

Most of the time, a computer has more runnable work than CPU cores. The operating system uses [scheduling](https://en.wikipedia.org/wiki/Scheduling_(computing)) to decide which **process** or **thread** should run next.

## The Scheduler

The **scheduler** is the part of the operating system (or runtime) that decides which **thread** or **process** runs at any given moment.

## Types of Schedulers

Depending on their scope, schedulers can be:

- **OS kernel schedulers** — juggle processes and threads directly, deciding which runs on which core and for how long.
- **Runtime schedulers** — manage user-level threads on top of OS threads. Go's runtime scheduler is an example, mapping goroutines onto OS threads without the kernel's involvement.

## Run Queues

Schedulers keep track of runnable work using [run queues](https://en.wikipedia.org/wiki/Run_queue):

- **Local run queue** (LRQ) — one per core, holds threads that most recently executed on that core.
- **Global run queue** (GRQ) — catches overflow and newly awakened work.

```
┌─────────────── CPU Core 0 ───────────────┐
│ Running: Thread A                        │
│ ┌──────────── Local Run Queue ─────────┐ │
│ │ [B] → [C] → [D]                      │ │
│ └──────────────────────────────────────┘ │
└──────────────────────────────────────────┘

┌─────────────── CPU Core 1 ───────────────┐
│ Running: Thread E                        │
│ ┌──────────── Local Run Queue ─────────┐ │
│ │ [F] → [G]                            │ │
│ └──────────────────────────────────────┘ │
└──────────────────────────────────────────┘

┌────────────── Global Run Queue ───────────┐
│ [H] → [I] → [J]                           │
└───────────────────────────────────────────┘

┌────────────── Blocked / Wait List ────────┐
│ [K] → [L]                                 │
└───────────────────────────────────────────┘
```

How the scheduler manages these queues:

- **New task** — inserted into a run queue.
- **Task finishes** — scheduler pops the next entry from the local run queue and dispatches it to the CPU.
- **Task blocks** — removed from the run queue and parked on a wait list; returns to a run queue when the event completes.

## Preemptive Scheduling

[Preemptive scheduling](https://en.wikipedia.org/wiki/Preemption_(computing)) lets the scheduler interrupt a running task and hand the CPU to another, even if the running task hasn't blocked or yielded. This happens when:

- A **higher-priority** task becomes runnable.
- A **time slice** expires.

> [!NOTE]
> A [time slice](https://en.wikipedia.org/wiki/Preemption_(computing)#Time_slice) is the fixed chunk of CPU time granted to a task before it can be preempted. Short slices improve responsiveness; longer slices reduce context-switch overhead.

## Work Stealing

When a core's local run queue runs dry, the scheduler can **steal** tasks from another core's queue (or the global queue) to keep the CPU busy. Work stealing balances load across cores without routing every task through a single shared queue.
