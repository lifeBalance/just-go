# Error Groups

One of the shortcomings of WaitGroups is that don't have a built-in way to handle errors from goroutines. In case of error in any of the goroutines of the group, the `WaitGroup`: does not return values, propagate errors, or cancel the other goroutines.

For example, if three goroutines are fetching data and one of them returns an error, a plain `WaitGroup` has no built-in place to put that error. You have to store it yourself, usually with a channel or a shared variable protected by a mutex.

Fortunately for us, the [sync/errgroup](https://pkg.go.dev/golang.org/x/sync/errgroup) package provides synchronization, error propagation, and Context cancellation for groups of goroutines working on subtasks of a common task:

```go
func main() {
	if err := fetchDashboardData(context.Background()); err != nil {
		log.Fatal(err)
	}

	fmt.Println("dashboard data fetched successfully")
}

func fetchDashboardData(ctx context.Context) error {
	g, ctx := errgroup.WithContext(ctx)

	g.Go(func() error {
		return fetchUser(ctx)
	})

	g.Go(func() error {
		return fetchPosts(ctx)
	})

	return g.Wait()
}
```

`errgroup.WithContext` returns a group and a derived context. If any goroutine returns an error, that derived context is cancelled. The other goroutines can notice the cancellation through `ctx` and stop early, assuming they respect the context.

> [!WARNING]
> The problem with `errgroup` is that it only returns the first non-nil error. If you need to collect **all errors**, or protect shared data between goroutines, you'll need a manual approach: use a channel to gather all errors, or a `sync.Mutex` to safely write to a shared variable.

## Manual Approach

You can handle errors manually with a `WaitGroup`, but you need another synchronization mechanism to move errors back to the caller. One common approach is to use an error channel:

```go
func fetchDashboardData(ctx context.Context) error {
	var wg sync.WaitGroup
	errs := make(chan error, 2)

	wg.Add(2)

	go func() {
		defer wg.Done()

		if err := fetchUser(ctx); err != nil {
			errs <- err
		}
	}()

	go func() {
		defer wg.Done()

		if err := fetchPosts(ctx); err != nil {
			errs <- err
		}
	}()

	wg.Wait()
	close(errs)

	for err := range errs {
		if err != nil {
			return err
		}
	}

	return nil
}
```

This works, but notice how much more code we need:

- The `WaitGroup` waits for the goroutines.
- The `errs` channel carries errors back to the caller.
- We close the channel after all goroutines finish.
- We still have to decide what to do with multiple errors.

This is why `errgroup` is often preferred when each goroutine can fail. It packages the common pattern into one tool: start goroutines with `g.Go`, return errors from each goroutine, and call `g.Wait()` to wait for completion and receive the first error.
