# Wait Groups

A [sync.WaitGroup](https://pkg.go.dev/sync#WaitGroup) allows us to wait for a known number of goroutines to finish. A `Waitgroup` doesn't synchronize or protect data, just coordinate execution.

## Waiting For Goroutines

When you launch a goroutine, the main function doesn't wait for it to finish. If the main function exits before the goroutine completes, the program will terminate and the goroutine will be abruptly stopped. Check this out:

<Code code={wgNeeded} lang='go' title='wg-needed/main.go'/>

If we run the code above, we'll get an output like this:

```
All workers done
```

The main function exits before any of the worker goroutines have a chance to print their messages. To prevent this, we can use a `WaitGroup` to ensure that the main function waits for all goroutines to finish before exiting:

<Code code={wg} lang='go' title='wg/main.go'/>

If we run the code above, we'll get the expected output:

```
Worker 2 starting
Worker 1 starting
Worker 3 starting
Worker 3 done
Worker 1 done
Worker 2 done
All workers done
```

As you can see, the main function waits for all worker goroutines to complete before printing "All workers done" and exiting. But the order of the worker messages may vary each time you run the program, since goroutines are scheduled independently by the **Go runtime**.

> [!WARNING]
> As you can see in the code and output above, `wg.Wait()` blocks the current goroutine (in this case `main`) until every tracked goroutine has called `wg.Done()`.
>
> To clarify: `wg.Wait()` does not block every goroutine in the program. It only pauses the goroutine where `wg.Wait()` is called (in this case `main`).

## What WaitGroups Do Not Do

In the code above, in `main`:

2. We tell the `WaitGroup` to wait for 2 goroutines (`wg.Add(2)`).
3. We launch the two goroutines.
4. We wait for the group to finish. `wg.Wait()` - blocks until counter reaches zero (all goroutines done)

If we run this code, we'll still get inconsistent results:
```
Balance: 8459490
Balance: 5821070
Balance: -9993470
```

We just got rid of the `time.Sleep` call, so that `main` goroutine will wait for the group to finish. But we're not controlling the way the goroutines in the group access the `money`.

<Admonition type="warning" title="⚠️ A WaitGroup just waits">
`WaitGroup` only ensures we **wait** for goroutines to finish—it does NOT protect **shared data**.
</Admonition>

## Combining With Other Synchronization Tools

WaitGroups are often used in combination with other synchronization tools, like mutexes or channels, to coordinate the execution of goroutines while also ensuring safe access to shared resources.

Some examples:

- [WaitGroup + Mutex](/concurrency/synchronization/mutexes#the-art-of-mutexing): Use a `WaitGroup` to wait for goroutines to finish and a `Mutex` to protect shared data.
- [WaitGroup + Channel](/concurrency/synchronization/channels#multiple-owner): Use a `WaitGroup` to wait for goroutines to finish and a `Channel` to communicate results or signals between goroutines.

If your goroutines can fail and you want to stop the whole group when one returns an error, use `errgroup`, which builds on the same idea but adds error handling and cancellation.
