# Concurrency in Go

Go has **first-class support for concurrency** built into the language itself. Rather than exposing [OS threads](https://en.wikipedia.org/wiki/Thread_(computing)) directly, Go provides its own lightweight abstraction: the **goroutine**.

> [!NOTE]
> C, C++, Java, and Rust all expose OS threads directly (via `pthread`, `std::thread`, `java.lang.Thread`, etc.). You create a thread, the OS manages it, and you pay the full OS cost per thread.

Goroutines are managed by the Go runtime, not the OS. They start at around `2KB` of stack space (compared to `~1MB` for an **OS thread**) and are much cheaper to create and switch between — you can spawn thousands, or even millions, in a single program. A built-in [scheduler](https://go.dev/src/runtime/HACKING) automatically distributes them across available CPU cores.

Go provides four primitives for concurrent programming:

- [Goroutines](https://go.dev/tour/concurrency/1): Lightweight runtime-managed threads. The fundamental unit of concurrent execution in Go.
- [Channels](https://go.dev/tour/concurrency/2): Typed pipes for passing data between goroutines safely, without shared memory.
- [Select](https://go.dev/tour/concurrency/5): Lets a goroutine wait on multiple channel operations at once — like a switchboard for concurrent data flows.
- [Sync Package](https://pkg.go.dev/sync): Low-level synchronization tools like `WaitGroup` (wait for a group of goroutines to finish) and `Mutex` (protect shared memory from concurrent access).
